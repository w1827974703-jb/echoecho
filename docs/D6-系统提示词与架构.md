# D6 · AI 出题 + AI 情景短文：System Prompt 与架构

> 记录 D6 两个模块（`/api/quiz` 出题、`/api/story` 情景短文）的**系统提示词原文**与**技术架构**。
> 关键认知：D6 **不是 Agent**、**不用 function calling / 工具**、**不做多轮编排**——是「单次结构化生成 + 强 prompt 约束 + 后端兜底解析」。
> 对照：D4 生词模块（`/api/word`）才是真 Agent（用了两个工具 + 多轮 function calling）。

---

## 一、AI 出题（`/api/quiz`）

- **入参**：`{ words: {word, meaning, sentence}[] }`（≤10）
- **出参**：`{ questions: Question[] }`（每词 1 题、4 选项、1 正确项、中文解析）
- **源码**：`src/app/api/quiz/route.ts`

### System Prompt（原文）

```text
你是英语出题助手。根据给定的生词列表出选择题，用来帮助学习者巩固记忆。
严格要求：
1. 只出选择题；每个词出且仅出 1 题，题量 = 词数。
2. 每题恰好 4 个选项，其中 1 个正确；干扰项要合理、有迷惑性，不要明显荒谬。
3. stem 是题干（可用该词的原句挖空或造情景），explanation 是中文解析（简短，说清为什么选它）。
4. answerIndex 是正确选项在 options 中的下标（0-3）。
5. word 字段填对应的目标生词。
最终只输出 JSON，形如：{"questions":[{"word":"...","stem":"...","options":["..","..","..",".."],"answerIndex":0,"explanation":"..."}]}
不要输出 JSON 以外的任何文字，不要用 markdown 代码块包裹。
```

### User Prompt（拼接模板）

```text
生词列表（共 {N} 个，请出 {N} 道选择题）：
1. {word}｜语境义：{meaning}｜原句：{sentence}
2. ...
```

## 二、AI 情景短文（`/api/story`）

- **入参**：`{ words: string[] }`
- **出参**：`{ story, targets, gloss }`（4–6 句英文短文；targets = 实际用到的目标词；gloss = 中文大意）
- **源码**：`src/app/api/story/route.ts`

### System Prompt（原文）

```text
你是英语写作助手。用给定的目标生词，写一段自然的英文短文，帮助学习者在情景中记忆这些词。
严格要求：
1. 短文 4–6 句，主题贴近播客/日常，难度适中、自然连贯。
2. 尽量把所有目标词自然编进去；targets 列出实际用到的目标词（保持原词形，便于前端高亮）。
3. gloss 是这段短文的中文大意（简短，1–2 句）。
最终只输出 JSON，形如：{"story":"...","targets":["..",".."],"gloss":"..."}
不要输出 JSON 以外的任何文字，不要用 markdown 代码块包裹。
```

### User Prompt（拼接模板）

```text
目标生词：{word1, word2, ...}
```

## 三、Prompt 设计的三个要点（为什么这么写）

1. **角色 + 严格规则 + JSON 样例三件套**：先定角色（出题/写作助手），再逐条列硬规则，最后给一个 JSON 模板，让模型「照着填」。
2. **双重「只输出 JSON」约束**：既说"最终只输出 JSON"，又补一句"不要用 markdown 代码块包裹"——因为模型爱自作主张包 ```json。
3. **不完全信任模型**：prompt 只是软约束，真正的保证在后端——
   - `parseLooseJson()` 容忍噪声，从文本里抠出最外层 `{...}`；
   - quiz 兜底过滤不合格的题（选项≠4、answerIndex 越界）；
   - story 兜底补 `targets`（模型没给就用输入词在正文里出现的）。

## 四、技术架构（简单可视化）

### 分层总览

```
浏览器（巩固页 consolidate/page.tsx）
  · pickWords()：new/unknown 优先，不足用 known 补，≤10
  · [生成练习题] → POST /api/quiz  {words:[{word,meaning,sentence}]}
  · [生成短文]   → POST /api/story {words:[string]}
  · 结果存 sessionStorage（跳页/刷新不丢）
  · QuizPanel 判分 / StoryPanel 高亮+点词查释义
        │  只跟自己的 /api 通信，绝不直连阿里云
════════ 安全边界：DASHSCOPE_API_KEY 只在服务端 ════════
        ▼
Next.js API Route（quiz/route.ts、story/route.ts）
  ① 校验+清洗入参（去空 / 去重 / 截断 ≤10）
  ② 拼 system + user prompt
  ③ 调 lib/qwen.ts 的 generateJson()
  ④ 兜底清洗 → NextResponse.json(...)
        │
  lib/qwen.ts（共享底座，D4/D6 复用）
  · qwenModel() = createOpenAI({baseURL, apiKey}).chat("qwen-plus")
  · generateJson() = generateText() + parseLooseJson()
  · 读 process.env.DASHSCOPE_API_KEY
        │  HTTPS · Chat Completions 协议
        ▼
阿里云百炼 · 通义千问 qwen-plus
  OpenAI 兼容端点 /compatible-mode/v1
```

### 单次「生成练习题」时序（story 同构）

```
点[生成练习题]
  → pickWords() 挑 ≤10 词
  → POST /api/quiz
     ① 解析 body → 清洗 words（filter 去空 + slice(0,10)）
     ② 空词 → 400
     ③ 拼成编号词列表
     ④ generateJson({system: 出题规则, prompt: 词列表})
          → generateText({model: qwenModel(), system, prompt})   ← 单次，无 tools、无多轮
          → Qwen 返回文本（可能带 ```json 噪声）
          → parseLooseJson() 抠出 {...} → JSON.parse
     ⑤ 兜底过滤：每题 4 选项、answerIndex∈[0,4)
     ⑥ 全被过滤 → 502；否则返回 {questions}
  → setQuestions() → 存 sessionStorage
  → QuizPanel 渲染：点选项即时判分（绿✓/红✗）+ 解析 + 得分
```

## 五、D4 vs D6 一图对比（吃透「是不是 Agent」）

| 维度 | D4 生词 `/api/word` | D6 出题/短文 `/api/quiz`、`/api/story` |
|------|--------------------|-----------------------------------------|
| 是不是 Agent | ✅ 是 | ❌ 不是 |
| function calling / 工具 | ✅ `lookupDictionary`、`translateInContext` | ❌ 无 |
| 调用方式 | `generateText` + `stopWhen: stepCountIs(6)` 多轮 | `generateText` 单次 |
| 模型行为 | 自己决定调哪个工具、多轮往返 | 一次读 prompt、一次吐 JSON |
| 共用底座 | `lib/qwen.ts` 的 `qwenModel()`（`.chat()` 接入） | 同左 |

**为什么 D6 故意不用 Agent**：出题/短文是一步能完成的确定性任务，不需要模型「查资料→决策→再行动」。上 Agent 只会更慢、更贵、更不稳。这是合理选型，不是偷懒。
